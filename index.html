Last Sunday, March 7, I sat down with Claude Code in the Terminal and built a todo app from scratch.

No IDE, no prior setup. I described what I wanted for daily tasks, long-term goals, a workout tracker, modular quadrants for language learning/habits/spending and spent the better part of an afternoon going back and forth until it worked.

By the end, it was live on GitHub Pages and I was using it every day.

That's how I kicked this off. Today I wanted to iterate on it: fix some things that had been bugging me, figure out how to properly set up VS Code, build a place to document this work, and get the app in front of real users.

I also did something I hadn't done before: before writing a line of code, I used Claude's Cowork tool to plan out what each task actually required. It helped me figure out the right prompts to take into Claude Code so I wasn't figuring things out from scratch each time, but honestly, it wasn't as useful as I thought. It had a hard time actually helping me plan the tasks; when it wrote code, it was good, but I wasn't looking for code. I did some research, and looks like a structured Claude.MD file is the best way to get actually useful help with planning/specific tasks, so will try that next time.

Building the App (March 7, Backstory)

The app isn't just a checklist. It tracks daily and long-term goals as separate lists, has a workout tracker that activates when you log an exercise-related task, and is organized in modular quadrants so you can toggle on a language learning tracker, a habit tracker, or a spending log depending on what you're trying to stay on top of.

Everything is stored locally, so your workout history and spending data persist across sessions, and it's usable on both phone and desktop, though the local storage doesn't pass across the two.

One thing I'm particularly glad I did: before shipping, I asked Claude to propose a test suite, we aligned on what good coverage looked like, and then I used Playwright to run automated tests through the onboarding flow.

Playwright essentially plays the role of a user and walks through every step, filling in fields, clicking through screens, and confirming the right things appear. It's a tool engineers use to test web apps, and being able to apply it here let me ship with more confidence that the app actually worked.

This Week — Fixing What Wasn't Working

When I started using the app daily, two problems became obvious.

First: the mobile UI wasn't sharp. Text was a little small, tap targets cramped, spacing that worked on desktop felt off on a phone. I had downloaded VS Code specifically so I could get into the HTML and CSS myself for this kind of thing rather than prompting Claude for every small tweak and burning through my usage on edits I could just make directly.

I read through the stylesheet, understood what each piece was doing, and tightened it up. There's something clarifying about editing your own code by hand; it's a different kind of understanding than reading a summary of what Claude changed.

To actually diagnose what was off, I used Playwright and Chromium to spin up a test instance of Chrome configured as an iPhone 15 — essentially a simulated mobile browser running locally.

From there I had Claude take screenshots of the app at that viewport and we could see the problem clearly: the header banner had text overflowing out of its container, wrapping awkwardly and looking unprofessional. Not something you'd catch on desktop.

We fixed it directly from the screenshots, re-ran, and confirmed it looked clean. That loop — simulate, screenshot, fix, verify — is the right way to work on mobile UI, and it's the same underlying tool I used to test onboarding on March 7. Playwright keeps coming up.

The second problem was a missing feature. I kept going back to iPhone Notes to plan my weekends — one note, three sections (Friday, Saturday, Sunday), tasks and events I'd check off as the weekend moved. My girlfriend does a similar thing, but in her Reminders app, and I wanted us both to have something more robust.

So I'm adding a Weekend view: a persistent three-section checklist that lives outside the daily task flow. I drafted the Claude Code prompt for it today and worked to get it up and running, and am so far pleased with the results.

VS Code + Claude Code

I had Claude Code running in the Terminal already, so today was also about upgrading the setup. The VS Code extension puts Claude in a panel docked to your IDE, which is meaningfully better than switching back and forth to a terminal window.

Two things I'll use consistently going forward.

Plan Mode: before Claude touches any code, it writes out what it intends to do in plain language. You can read it, push back, annotate it. That's the right model for the kind of work I'm doing — I want to make the architectural calls, not just approve outputs.

And CLAUDE.md: a file at the root of each project that Claude reads every session. Coding conventions, how to run tests, what to check on mobile. You write it once and stop repeating yourself.

The division I've landed on: Cowork for planning and drafting prompts, Claude Code for building, VS Code for the edits I can and should make myself.

Getting the App Out There

The app has been private long enough. Before sharing it, I wanted the URL to not look like a side project someone threw together.

The original link — rcorson2018-star.github.io/todo-app — has my old GitHub username in it and a generic slug. To clean it up, I created a GitHub organization called ROCOLabs, transferred the repo there, and updated the local remote to point to the new location.

The app now lives under the ROCOLabs org, with a URL that actually looks like something. GitHub handles the redirect from the old link automatically.

The message I'm sending is deliberately low-key — "I built this, it might have rough edges, tell me what you find" rather than a product announcement. Early users with zero stakes are the best feedback you can get.

The through-line in all of this: I'm not an engineer, but I can engage with a codebase, make decisions about what to build, ship independently, and iterate based on what I find. That's what this log is for — to show that process as it actually happens.
